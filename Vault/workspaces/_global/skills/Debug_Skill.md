# 🐛 除錯流程 Skill

> 載入此技能包以確保除錯過程系統化、不走彎路。

---

## 黃金法則

1. **每次只改一個變數**（Single Change Principle）
2. **先重現問題，再修改**
3. **逐步收縮法**：從大範圍縮小到最小可重現案例
4. **記錄假設**：明確寫下「我認為問題在 X」再驗證
5. **二分搜尋**：不確定範圍時，先排除一半

---

## 標準除錯流程

```
1️⃣ 重現問題
   └─ 確認可穩定重現 → 記錄重現步驟

2️⃣ 縮小範圍（逐步收縮法）
   ├─ 問題在哪個層？（UI / 業務邏輯 / 資料 / 環境）
   ├─ 何時開始出現？（最近哪個 commit 引入？)
   └─ 最小重現案例是什麼？

3️⃣ 建立假設
   └─ 明確陳述：「我認為 X 導致 Y 因為 Z」

4️⃣ 驗證假設
   ├─ YES → 找到根因 → 修復
   └─ NO  → 回到步驟 2（排除一個方向）

5️⃣ 修復 + 驗收
   ├─ 修復後確認原始問題消失
   ├─ 確認沒有引入新問題（回歸測試）
   └─ 記錄根因 + 解法（避免重蹈覆轍）
```

---

## 常見錯誤陷阱

| 陷阱 | 後果 | 正確做法 |
|------|------|----------|
| 同時改多個地方 | 不知道哪個改動有效 | 每次只改一處 |
| 沒重現就開始改 | 修改無效後更混亂 | 先穩定重現 |
| 靠猜測亂改 | 浪費時間、可能引入新問題 | 先建立假設再驗證 |
| 修完就結束 | 問題可能再次出現 | 記錄根因 |

## Python / C# 常用除錯工具

```python
# Python：最小化範圍
import pdb; pdb.set_trace()      # 中斷點除錯
print(repr(variable))            # 確認型別和值

# 二分搜尋法找 bug
# 1. 註解掉後半段 → 問題消失？ → bug 在後半段
# 2. 恢復後半段、註解前半段 → 繼續縮小
```

```csharp
// C# Unity：最小化範圍  
Debug.Log($"[Debug] value={value}, type={value.GetType()}");
// 使用條件斷點而非 print flood
```
