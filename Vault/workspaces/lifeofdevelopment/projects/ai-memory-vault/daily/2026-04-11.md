### Session 2（午～晚）— Frozen exe torch 崩潰修復
- **根因分析**：`import torch` 在 PyInstaller 凍結環境中觸發 STATUS_STACK_BUFFER_OVERRUN（0xC0000409）
- **發現兩條 torch 匯入鏈**：
  1. `langchain_huggingface` → `sentence_transformers` → `import torch`（嵌入模型路徑）
  2. `EnsembleRetriever` → `langchain_classic` → `MultiQueryRetriever` → `transformers.utils.generic` → `import torch`（搜尋路徑）
- **解法 Layer 1 — Torch Stub**（main.py）：`_TorchStub(types.ModuleType)` 動態 stub，11 個子模組註冊至 `sys.modules`，攔截所有 `import torch` 呼叫，返回 `_FakeDtype()` 佔位
- **解法 Layer 2 — _OnnxEmbeddings**（embeddings.py）：完全繞過 `sentence_transformers`，使用 `onnxruntime + tokenizers + huggingface_hub` 直接推理，嵌入精度誤差 < 4.8e-7
- **修復 _FakeDtype NameError**：`del _FakeDtype` 誤刪運行時需要的類別 → 改為保留
- **修復 search_vault TextSplitter 匯入**：移除舊的假 `langchain_text_splitters` 套件 hack（torch stub 已確保不崩潰）
- **修復 search_instincts unpack 錯誤**：`VaultService.search()` 回傳 dict 列表，非 tuple → 改用 dict 存取
- **全面驗證**：建立 40 工具測試腳本（test_exe_mcp.py），45 步驟全通過，40/40 工具覆蓋率 100%

## 遇到的問題

1. `_FakeDtype` NameError — `del` 語句刪除了 `_TorchStub.__getattr__` 運行時仍需的類別
2. TextSplitter 匯入失敗 — 舊的假套件 hack 與 torch stub 方案衝突
3. InstinctService.search() unpack 錯誤 — API 回傳型別假設錯誤（以為是 tuple，實際是 dict）
5. 對話數重複計算 — `iProjectConvs` 含 detail 檔名，Template 未排除
6. 評分邏輯不合理 — `_auto_score` 問題解決率用 `success_rate/20` 概念錯誤、Prompt 技巧因 avg_qa=0.6 給假高分
7. **Scheduler 進入點缺少 _bootstrap** — `_start_scheduler_task()` 等函式直接建立 `AutoScheduler`，未初始化 vectorstore/embeddings，導致寫入操作失敗

## 筆記 / 學到的事

- **Torch stub 設計要點**：`__getattr__` 必須先查 `sys.modules` 再回退 FakeDtype，否則子模組屬性存取會被攔截
- **不要 del stub 類別**：被 `__getattr__` 閉包引用的類別不能 del，否則運行時 NameError
- **一旦有全域 stub，舊的局部 import hack 就多餘了**：舊的 `_get_splitters()` 假套件 hack 反而會與 stub 衝突
- **API 合約要驗證**：`VaultService.search()` 回傳 dict 列表，但 `InstinctService` 假設是 tuple，服務間介面要明確
- **detail 解析設計**：只用 Markdown 結構解析（## / ### / 表格行 / 列表），不依賴 frontmatter，鬆耦合
- **平均值需區分有無資料**：無 QA 記錄的對話不納入平均輪數計算，避免拉低數字
- **錯誤模式要帶上下文**：只存 pattern+count 不夠，要帶 cause/solution 才有分析價值
- **所有排程進入點都需要 _bootstrap**：`_start_scheduler_task()`/`_start_scheduler_once()`/`_start_scheduler()` 與 MCP/CLI 模式不同，它們直接建立 AutoScheduler，必須先初始化 vectorstore/embeddings
- **冪等機制掩蓋 bug**：weekly-ai 成功只因檔案已存在、提前返回，未觸發寫入路徑；monthly-ai 需要寫入才暴露問題
- **build.spec 排除策略**：除了不 collect_all，還要在 excludes 明確列出套件名，雙重保險防止被其他套件拖入
